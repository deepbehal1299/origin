---

## 1. Contract Version

- **Version:** 1.0
- **Scope:** Version 1 only
- **Authentication:** None
- **Primary API surface:** `GET /coffees`

### v1 constraints

- No user accounts or login
- No checkout or payment APIs
- No notifications, recommendations, or social features
- No query params on `GET /coffees` in v1
- Frontend filtering is client-side in v1

## 2. Canonical Domain Types

These types define the exact values and shapes expected across the backend API, frontend UI, scraper normalization logic, and tests.

### 2.1 Roast level enum

```ts
export type RoastLevel =
  | "Light"
  | "Light-Medium"
  | "Medium"
  | "Medium-Dark"
  | "Dark";
```

**Rules:**

- Only these exact strings are allowed.
- Scrapers must normalize source values to one of these strings or `null`.
- Frontend filters and roast preferences must use these exact strings.

### 2.2 Supported roasters

These are the exact display names used in Settings, Feed filters, and coffee payloads.

```ts
export type RoasterName =
  | "Subko"
  | "Savorworks"
  | "Bloom Coffee Roasters"
  | "Rossette Coffee Lab"
  | "Marcs Coffee"
  | "Grey Soul Coffee";
```

Recommended stable ids for config, DB, and filtering logic:

| Display name | Stable id |
|---|---|
| Subko | `subko` |
| Savorworks | `savorworks` |
| Bloom Coffee Roasters | `bloom-coffee-roasters` |
| Rossette Coffee Lab | `rossette-coffee-lab` |
| Marcs Coffee | `marcs-coffee` |
| Grey Soul Coffee | `grey-soul-coffee` |

### 2.3 Coffee type

```ts
export interface Coffee {
  id: string;
  name: string;
  roaster: RoasterName | string;
  roaster_id: string;
  roast_level: RoastLevel | null;
  tasting_notes: string | null;
  description: string | null;
  price: number;
  weight: string | null;
  image_url: string | null;
  product_url: string;
  available: boolean;
}
```

### 2.4 Field semantics

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | `string` | Yes | Unique coffee identifier generated by backend |
| `name` | `string` | Yes | Coffee product name |
| `roaster` | `string` | Yes | Exact roaster display name |
| `roaster_id` | `string` | Yes | Stable backend/config identifier |
| `roast_level` | `RoastLevel \| null` | Yes | `null` when source data cannot be confidently normalized |
| `tasting_notes` | `string \| null` | Yes | Source notes preserved as a single string if structure varies |
| `description` | `string \| null` | Yes | Origin, process, varietal, or product description |
| `price` | `number` | Yes | Numeric product price for the selected variant |
| `weight` | `string \| null` | Yes | Example: `200g`, `250g`, `500g` |
| `image_url` | `string \| null` | Yes | Absolute image URL when available |
| `product_url` | `string` | Yes | Absolute URL to the roaster's product page |
| `available` | `boolean` | Yes | API returns only available coffees in v1 |

### 2.5 Type notes

- `price` must be numeric in the API response, not a formatted currency string.
- Nullable fields must be returned as `null` when unavailable, not omitted.
- `product_url` and `image_url` should be absolute URLs.
- `roaster` should use the canonical display names above.
- Although `roaster` is constrained to the supported v1 list, the interface permits `string` so the app can evolve without forcing a breaking type rewrite immediately.

## 3. Endpoint Contract

### 3.1 `GET /coffees`

- **Method:** `GET`
- **Path:** `/coffees`
- **Auth:** None
- **Query params:** None in v1
- **Purpose:** Return the normalized list of currently available coffees aggregated from configured roasters

### Success response

- **Status:** `200 OK`
- **Content-Type:** `application/json`
- **Body type:**

```ts
export type GetCoffeesResponse = Coffee[];
```

### Response example

```json
[
  {
    "id": "5a7a7ac4-bfce-49cb-8f89-e3d9608b7c8d",
    "name": "Ratnagiri Estate",
    "roaster": "Subko",
    "roaster_id": "subko",
    "roast_level": "Light-Medium",
    "tasting_notes": "stone fruit, cacao nib, citrus",
    "description": "Single origin coffee with washed processing from Ratnagiri Estate.",
    "price": 850,
    "weight": "250g",
    "image_url": "https://subko.com/cdn/shop/files/ratnagiri.jpg",
    "product_url": "https://subko.com/products/ratnagiri-estate",
    "available": true
  }
]
```

### Behavioral guarantees

- Response body is always an array.
- If no coffees are available, response is `[]`.
- Only coffees with `available === true` are returned. The field is included in every response object for transparency; it will always be `true` in v1 API responses.
- Every object in the array must conform to the `Coffee` type above.
- The frontend is responsible for roaster and roast filtering in v1.

### Non-goals for this endpoint in v1

- No pagination
- No sorting params
- No server-side filtering
- No search
- No write operations
- Requests to other paths or unsupported methods return a backend-default 404 or 405

## 4. Error and Empty-State Expectations

The plans define only a single read endpoint for v1, so error handling should remain simple and predictable.

### Expected behaviors

- `200` with `[]` when there are no available coffees
- `5xx` only for genuine backend failure conditions
- No custom error schema is required for v1, but JSON responses are preferred if errors are returned

### Frontend expectation

- The UI must handle an empty array gracefully
- The UI must not assume optional fields are populated
- The UI should tolerate temporary API failure and show an error or retry state without breaking navigation

## 5. CORS Contract

Backend must allow the frontend origin to call `GET /coffees`.

### Development

- Frontend origin: `http://localhost:3000`
- Backend default port: `4000` (set `NEXT_PUBLIC_API_URL=http://localhost:4000` in development)

### Production

- Allowed origin: deployed frontend/PWA origin

### Minimum expectation

- Allow `GET`
- Allow `Content-Type` header as needed
- Browser requests from the configured frontend origin must succeed without CORS blocking

## 7. Client Persistence Contract

These are not backend API payloads, but they are part of the implementation contract shared by Feed, Compare, Settings, and tests.

### 7.1 Compare state

```ts
export type CompareStorageValue = string[];
```

| Key | Type | Rules |
|---|---|---|
| `origin_compare` | `string[]` | Max length 5 — store ids, resolve from API data |

**Rules:**

- Store coffee `id` values only. Resolve the full objects from the fetched API data when rendering the Compare page.
- Storing full objects risks stale data if the backend rescrapes with updated prices or fields.
- App must never persist more than 5 items.
- Compare page must support reload persistence from this key.

### 7.2 Roaster settings

```ts
export type RoasterSettingsStorageValue = Record<string, boolean>;
```

| Key | Type | Rules |
|---|---|---|
| `origin_roasters` | `Record<string, boolean>` | Maps roaster display name to enabled state |

**Default:** If the key is absent in localStorage, all roasters are treated as enabled.

Example:

```json
{
  "Subko": true,
  "Savorworks": false,
  "Bloom Coffee Roasters": true
}
```

### 7.3 Roast preferences

```ts
export type RoastPreferencesStorageValue = RoastLevel[];
```

| Key | Type | Rules |
|---|---|---|
| `origin_roast_preferences` | `RoastLevel[]` | Uses exact roast enum strings |

**Default:** If the key is absent or the array is empty, no roast filter is applied and all roast levels are shown.

### 7.4 Saved coffees

The PRD includes a Save action on coffee cards, and the test cases document a local bookmark contract.

```ts
export type SavedStorageValue = string[];
```

| Key | Type | Rules |
|---|---|---|
| `origin_saved` | `string[]` | Client-only bookmark persistence; store ids; no backend sync in v1 |

## 9. Contract Alignment Checklist

Use this checklist before implementation or review:

- `GET /coffees` returns `200` and a JSON array
- Every coffee object matches the documented `Coffee` shape
- `price` is numeric, not formatted text
- `roast_level` only uses the exact supported enum strings or `null`
- `roaster` uses the canonical display names
- API includes only available coffees
- Frontend uses `NEXT_PUBLIC_API_URL`
- `origin_compare` never exceeds 5 items
- `origin_roasters` stores roaster name to boolean
- `origin_roast_preferences` stores exact roast enum strings
- `origin_saved` is client-only and has no backend dependency in v1

---

## 10. Change Policy

Any future change to the `Coffee` shape, roast enum values, roaster names, localStorage keys, or `GET /coffees` behavior should update:

- this contract document
- the LLD if backend behavior changes
- `docs/test-cases.md` if acceptance criteria change

This keeps implementation, review, and testing aligned from a single contract baseline.
